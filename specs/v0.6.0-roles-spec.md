**Settlement Flow Detail:**

1. The Valuation Manager (or Security Council) proposes a `newTotalAssets` via `updateNewTotalAssets()`
2. This triggers an epoch change (freezes pending requests)
3. The Safe calls `settleDeposit()` or `settleRedeem()` passing the same `newTotalAssets`
4. Internally: `updateTotalAssets()` verifies consistency, takes fees, then settles the requests

### 1.4 Valuation Manager

**Definition:** `Roles.sol:69-72` -> `RolesLib._onlyValuationManager()` (`RolesLib.sol:42-47`)
**Modifier:**   `onlyValuationManager`
**Storage:**    `RolesStorage.valuationManager`

Responsible for updating the vault's Net Asset Value (NAV). Subject to guardrails.

#### Actions

| # | Function                        | File:Line              | Conditions | Description                                |
|---|---------------------------------|------------------------|------------|--------------------------------------------|
| 1 | `updateNewTotalAssets(uint256)` | `Vault-v0.6.0.sol:435` | See below  | Proposes a new NAV valuation for the vault |

**Detailed Conditions for `updateNewTotalAssets()`:**

1. Vault must **not** be `Closed` -> reverts `Closed()`
2. `totalAssets` must be **expired** (async mode) -> reverts `ValuationUpdateNotAllowed()` if still valid
3. Contract must **not** be paused (checked in `ERC7540Lib.updateNewTotalAssets`)
4. **Guardrails check** (skipped if `lastFeeTime == 0`, i.e., first update —
   however `__FeeManager_init` sets `lastFeeTime = block.timestamp`, so this branch
   is unreachable after proper initialization; the check exists in code but is effectively dead):
   - Computes current PPS: `convertToAssets(10^decimals)`
   - Computes proposed PPS: `10^decimals * (newTotalAssets + 1) / (totalSupply + 10^decimalsOffset)` (ERC4626 virtual offset pattern)
   - Compares against upper/lower guardrails limits
   - Reverts `GuardrailsViolation()` if non-compliant

**Side Effects of `updateNewTotalAssets()`:**

- Freezes pending deposit/redeem requests (epoch change)
- Saves `pendingAssets` and `pendingShares` in `SettleData`
- Stores `newTotalAssets` for the Safe to use during settlement

### 1.6 Fee Receiver

**Definition:** `Roles.sol:25` (field `RolesStorage.feeReceiver`)
**Modifier:**   None (passive role)
**Storage:**    `RolesStorage.feeReceiver`

Recipient of fees generated by the vault.

#### Behavior

- Automatically receives minted shares on each fee collection via `FeeLib.takeFees()` -> `ERC7540.forge()`
- Fees are split between the Fee Receiver (manager shares) and the Protocol Fee Receiver
  (protocol shares) according to the `protocolRate`
- **Note on access control:** `AccessableLib.isAllowed()` (`AccessableLib.sol:131-133`) bypasses
  checks for the **Protocol Fee Receiver**, not the vault-level Fee Receiver

**Fee Types Received:**

| Type        | Collection Moment                                                       | Max Rate              |
|-------------|-------------------------------------------------------------------------|-----------------------|
| Management  | At each settlement (pro-rata temporis)                                  | 10% annual (1000 BPS) |
| Performance | At each settlement (above HWM)                                          | 50% (5000 BPS)        |
| Entry       | At deposit settlement and sync deposits                                 | 10% (1000 BPS)        |
| Exit        | At redeem settlement, sync redeems, and withdraw/redeem on Closed vault | 10% (1000 BPS)        |
| Haircut     | On sync redeems only (burned, not redistributed)                        | 10% (1000 BPS)        |

Global operator that can act on behalf of **any** user without requiring individual approval.

#### Actions

| # | Function                                   | File:Line              | Conditions                 | Description                                                                                  |
|---|--------------------------------------------|------------------------|----------------------------|----------------------------------------------------------------------------------------------|
| 1 | `deposit(uint256, address, address)`       | `ERC7540.sol:352`      | —                          | Claims a settled deposit for any controller                                                  |
| 2 | `mint(uint256, address, address)`          | `ERC7540.sol:383`      | —                          | Mints shares for any controller                                                              |
| 3 | `cancelRequestDeposit(address)`            | `ERC7540.sol:412`      | Not paused                 | Cancels a deposit request for any controller                                                 |
| 4 | `withdraw(uint256, address, address)`      | `Vault-v0.6.0.sol:293` | Not paused                 | Claims a withdrawal (async if claimable, sync if Closed)                                     |
| 5 | `redeem(uint256, address, address)`        | `Vault-v0.6.0.sol:329` | Not paused                 | Claims a redeem (async if claimable, sync if Closed)                                         |
| 6 | `transferFrom(address, address, uint256)`  | `ERC7540.sol:241`      | —                          | Transfers shares bypassing whitelist/blacklist in Blacklist mode, without consuming allowance|
| 7 | `requestRedeem(uint256, address, address)` | `Vault-v0.6.0.sol:262` | Vault `Open`, not paused   | Requests an async redeem for any controller                                                  |

#### Special Privileges

1. **Bypass whitelist/blacklist**: The super operator is always `isAllowed` (`AccessableLib.sol:136-139`)
2. **Bypass whitelist in operations**: When the super operator initiates `_deposit`, `_mint`,
   `_redeem`, `_withdraw` (claim), `_requestRedeem`, or `_withdraw` (Closed vault), `isAllowed` checks
   on owner/receiver/controller/msg.sender are skipped
3. **Bypass allowance**: In `transferFrom`, the super operator does not consume allowance (`ERC7540.sol:255`)
4. **Bypass blacklist on transferFrom**: In Blacklist mode, the super operator bypasses `isAllowed`
   checks (`ERC7540.sol:248`)

#### Restrictions

- **Cannot** act as super operator for the **Protocol Fee Receiver**
  (`RolesLib.sol:113`: `controller != _protocolFeeReceiver()`)
- **`cancelRequestDeposit(controller)`**: the super operator can call this function via
  `onlyOperatorOrSuperOperator`, but **cannot** bypass the `isAllowed(controller)` check inside
  `ERC7540Lib.cancelRequestDeposit` — if the controller is not allowed, the call reverts.
  This differs from `_deposit`, `_mint`, `_redeem`, `_withdraw`, and `_requestRedeem` which all
  have a `isSuperOperator` bypass for `isAllowed` checks.

2. wait "delay" seconds (the old delay)

## 4. Access Control (Whitelist / Blacklist)

### Modes (`AccessMode` enum)

| Value | Mode          | `isAllowed()` Logic                    |
|-------|---------------|----------------------------------------|
| `0`   | **Blacklist** | Address is allowed unless blacklisted  |
| `1`   | **Whitelist** | Address is allowed only if whitelisted |

### `isAllowed()` Check (`AccessableLib.sol:126-157`)

```
1. If account == protocolFeeReceiver        -> true (always)
2. If account == superOperator              -> true (always)
3. Whitelist mode: isWhitelisted[account]
4. Blacklist mode: !isBlacklisted[account]
5. If externalSanctionsList is set:         !isSanctioned(account)
6. Returns (internalListApproval && externalListApproval)
```

### Where `isAllowed()` is Checked

| Operation                       | Checks Performed                                                 |
|---------------------------------|------------------------------------------------------------------|
| `requestDeposit`                | owner, controller, msg.sender                                    |
| `_deposit` / `_mint` (claim)    | controller, receiver, msg.sender (skipped if super operator)     |
| `_requestRedeem`                | owner, controller, msg.sender (skipped if super operator)        |
| `_redeem` / `_withdraw` (claim) | controller, receiver, msg.sender (skipped if super operator)     |
| `_withdraw` (Closed vault)      | owner, receiver, msg.sender (skipped if super operator)          |
| `syncDeposit`                   | msg.sender, receiver                                             |
| `syncRedeem`                    | msg.sender, receiver                                             |
| `transfer`                      | msg.sender, to (Blacklist mode only)                             |
| `transferFrom`                  | from, to, msg.sender (Blacklist mode, skipped if super operator) |
| `claimSharesOnBehalf`           | controller, msg.sender (Safe) — via internal `_deposit`          |
| `claimAssetsOnBehalf`           | controller, msg.sender (Safe) — explicit + via internal `_redeem`|
| `claimSharesAndRequestRedeem`   | msg.sender                                                       |
| `cancelRequestDeposit` (1-arg)  | controller                                                       |
| `cancelRequestDeposit` (0-arg)  | msg.sender                                                       |

### Upgrade Compatibility v0.5.0 -> v0.6.0

The storage layout is compatible:

- Old `bool isActivated` at slot 1: `false (0)` -> `AccessMode.Blacklist (0)` = vault remains accessible
- Old `bool isActivated` at slot 1: `true (1)` -> `AccessMode.Whitelist (1)` = vault remains in whitelist mode

## 5. Fee System

### `Rates` Structure

```solidity
struct Rates {
    uint16 managementRate;   // max 1000 (10%)
    uint16 performanceRate;  // max 5000 (50%)
    uint16 entryRate;        // max 1000 (10%) — new in v0.6.0
    uint16 exitRate;         // max 1000 (10%) — new in v0.6.0
    uint16 haircutRate;      // max 1000 (10%) — new in v0.6.0
}
```

### When Fees Are Collected

| Type            | Moment                                                                                             | Mechanism                                                             |
|-----------------|----------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------|
| **Management**  | `settleDeposit()` / `settleRedeem()` / `close()`                                                   | Pro-rata temporis on `(previousTotalAssets + currentTotalAssets) / 2` |
| **Performance** | `settleDeposit()` / `settleRedeem()` / `close()`                                                   | Collected if PPS > High Water Mark                                    |
| **Entry**       | `settleDeposit()` (async), `syncDeposit()` (sync) ¹                                                | Fraction of minted shares                                             |
| **Exit**        | `settleRedeem()` (async), `syncRedeem()` (sync), `_redeem()` / `_withdraw()` (claim) ², closed vault | Fraction of burned shares                                             |
| **Haircut**     | `syncRedeem()` only                                                                                | Shares burned on top of exit fees (not redistributed)                 |

¹ At claim time (`_deposit` / `_mint`), the per-user entry fee deduction is applied against the
fee shares pre-minted during settlement — no additional `takeFees()` call occurs.

² Similarly, at claim time (`_redeem` / `_withdraw`), the per-user exit fee deduction is applied
against the fee shares pre-minted during `settleRedeem()` — no additional `takeFees()` call occurs.

### Fee Distribution

```
totalFeeShares = managementShares + performanceShares + entryShares + exitShares
protocolShares = totalFeeShares * protocolRate / 10000
managerShares  = totalFeeShares - protocolShares
```

- `managerShares`  -> minted to `feeReceiver` (vault level)
- `protocolShares` -> minted to `protocolFeeReceiver` (protocol level)
- `haircutShares`  -> burned (no one receives them)

**Note:** In practice, the protocol/manager split is applied **per-fee-type** (each `takeFees()` call
splits independently using `Math.Rounding.Ceil`), not on an aggregated total. Due to ceiling rounding,
the protocol may receive slightly more shares than the aggregated formula above suggests.

## 6. Guardrails

### Structure

```solidity
struct Guardrails {
    uint256 upperRate;  // Maximum annual PPS evolution rate (BPS)
    int256 lowerRate;   // Minimum annual PPS evolution rate (BPS, can be negative)
}
```

### How They Work

Guardrails limit the price-per-share (PPS) variation when `newTotalAssets` is updated
by the Valuation Manager.

- **Enabled/disabled** by the Security Council via `updateActivated(bool)`
- **Parameters** modified by the Security Council via `updateGuardrails(Guardrails)`
- **Not enforced** on the first update (`lastFeeTime == 0`)
- **Bypassed** by the Security Council via `securityCouncilUpdateTotalAssets()`
- **Check:** `isCompliant(currentPps, nextPps, timePast)` in `GuardrailsLib`
